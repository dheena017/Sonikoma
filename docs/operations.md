# Operations Guide

This document covers the operational aspects of running, monitoring, and troubleshooting the Sonikoma application in a production or long-running server environment.

## 1. Startup Process

The primary entry point for starting the application is:
```bash
npm run start
```
This single command orchestrates:
1. **Frontend**: Starts the Vite development/build server (`scripts/run-frontend.js`).
2. **Backend**: Starts the Python FastAPI application (`scripts/run-backend.js`), managing it as a child process and handling automatic restarts upon file changes.
3. **Environment Validation**: Both scripts strictly read the `.env` file and will fail-fast with a non-zero exit code if required variables (e.g., ports, API keys, JWT secret) are missing.

For production backend environments, use:
```bash
npm run start:prod
```
This runs the FastAPI app natively through Python without the development file-watcher overhead.

## 2. Configuration Management

All operational configuration must reside in the `.env` file at the repository root. Key variables:
- `FRONTEND_PORT`: Network port for the Vite app.
- `BACKEND_PORT`: Network port for the FastAPI server.
- `APP_URL`: The fully qualified URL where the frontend is hosted.
- `GEMINI_API_KEY`: Required AI provider token.
- `JWT_SECRET_KEY`: Cryptographic salt for generating session tokens.

The Python backend strictly validates these via `backend/app/core/config.py` on initialization.

## 3. Logging & Monitoring

### Standard Logging
The backend uses customized ANSI-colored logging mapped to standard output/error. `run-backend.js` intercepts these streams to present unified timestamps and severity levels (INFO, WARNING, ERROR, SUCCESS).

### Terminal Logs UI
Sonikoma includes an internal SSE (Server-Sent Events) mechanism. Logs generated by the Python backend are intercepted and streamed directly to the frontend's Terminal UI component for real-time visibility.

### Health Checks
The backend provides a `/api/health` HTTP GET endpoint. It returns a JSON payload detailing:
- System status (healthy/unhealthy).
- Process uptime.
- Database connection state.
- System memory usage (RAM).
Monitoring tools (like Docker healthchecks or Uptime Kuma) should poll this endpoint.

## 4. Database Backups

The application uses a local SQLite database (`data/webtoon_local.db`).
Because SQLite is a single file, backup and recovery simply involves copying this file.

**Backup Script/Process:**
1. Ensure the application is stopped to prevent file locks or partial writes.
2. Copy `data/webtoon_local.db` to a secure backup location.
3. (Optional) Zip the entire `data/` directory to back up downloaded image caches and final video renders.

## 5. Troubleshooting

**Common Issues:**
- **"Port already in use"**: Another process is binding the `FRONTEND_PORT` or `BACKEND_PORT`. Use `lsof -i :<port>` to find the PID and kill it.
- **Backend crashes on startup**: Check the terminal logs. It is almost always a missing variable in the `.env` file or a missing Python package (`pip install -r backend/requirements.txt`).
- **Video rendering fails**: Ensure `ffmpeg` is installed globally on the host operating system and is accessible via the system PATH.
